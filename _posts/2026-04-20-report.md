# Dev Diary #2 — From UI Skeleton to Extensible Runtime

## 1. Overview

지난 Dev Diary #1 에서는 디버그 대시보드의
**UI 구조 설계**에 초점을 맞추었다.

그러나 UI 구조만으로는 “실제 IoT 기기를 붙이는 단계”로 나아갈 수 없었다.
UI와 장치 로직이 한 파일(`run.py`) 안에 뒤섞여 있었고,
CV·하드웨어 제어 코드도 루트에 평평하게 흩어져 있었다.

이번 단계의 핵심 목표는 명확하다.

> **"UI는 그대로 두고, 그 아래 런타임 구조를 확장 가능한 형태로 재설계한다."**

본 다이어리에서는 이 재설계 과정에서 도입한
**3-Layer 구조 (devices / cv / hal)** 와
**장치 플러그인 시스템**을 정리한다.

---

## 2. What Changed Since #1

| 영역 | Diary #1 (2026-04-13) | Diary #2 (2026-04-20) |
|---|---|---|
| 장치 | 좌측 패널에 "라벨"만 존재 | `Device` ABC + 레지스트리 기반 플러그인 |

UI 계약(상태 5단계, 3-패널 레이아웃, 캐릭터 레이어)은
이번 리팩터에서도 그대로 유지되었다.
**"UI는 바뀌지 않지만, 그 아래는 전부 바뀐다"** 는 원칙이다.

---

## 3. Three-Layer Runtime

현재 런타임은 아래 세 개의 레이어로 분해되어 있다.

```id="layer-structure"
devices/   ← IoT 장치 플러그인 (교체·확장 대상)
cv/        ← 카메라·얼굴 DB·비전 파이프라인
hal/       ← 디스플레이 회전, 시스템 메트릭, BLE 스캐너
```

이 구분의 기준은 단순하다.

* **devices/** : *UI에 꽂히는 것* — 사용자가 자주 바꾼다
* **cv/** : *데이터를 만드는 것* — 모델/알고리즘이 바뀐다
* **hal/** : *OS와 맞닿은 것* — Pi 이식 시 집중 수정 지점

즉, "바뀌는 이유가 다르면 파일도 다르다" 는 SRP(단일 책임 원칙) 의
구체적 구현이다.

---

### 3.1 `devices/` — Plugin Layer

기존에는 `camera_panel.py`, `desk_fan.py` 가 루트에 있었고,
새 장치를 추가할 때마다 `run.py` 의 if-else 분기를 수정해야 했다.

이번 단계에서는 다음 구조로 전환했다.

```id="devices-tree"
devices/
├── __init__.py     ← 내장 장치 import
├── base.py         ← Device ABC
├── registry.py     ← @register 데코레이터 + 스텁 생성기
├── protocol.py     ← 공통 통신 프로토콜
├── transport.py    ← 공통 전송 계층
├── camera.py       ← 내장: 카메라
└── smart_fan.py    ← 내장: 서큘레이터
```

장치 구현체는 `Device` ABC 를 상속하고,
`@register("type_key")` 로 레지스트리에 자신을 등록한다.

```id="device-abc"
class Device(ABC):
    kind: ClassVar[str] = "other"   # bluetooth | serial | camera | other

    @classmethod
    @abstractmethod
    def from_config(cls, cfg: dict) -> "Device": ...

    @abstractmethod
    def make_panel(self) -> QWidget: ...

    def connect(self) -> None: ...
    def disconnect(self) -> None: ...
    def cleanup(self) -> None: ...
    def status(self) -> str: ...
```

`run.py` 는 이제 장치 타입을 **알지 못한다**.
단지 config 엔트리를 넘겨 레지스트리에서 인스턴스를 받아올 뿐이다.

```id="runtime-build"
for cfg in device_cfgs:
    dev = devices_registry.build(cfg)
    if dev is None:
        continue                        # 미등록 타입 → 기본 페이지
    panel = dev.make_panel()
    self._center.register_panel(dev.id, panel)
    dev.connect()
```

이 구조의 핵심은 **"새 장치 추가 시 수정할 파일의 수를 최소화"** 하는 것이다.

---

### 3.2 `cv/` — Computer Vision Module

기존에는 `face_db.py`, `vision.py`, `camera_source.py` 가
루트에 평평하게 놓여 있었다.

이 구조의 문제는 두 가지였다:

* **역할이 드러나지 않음** — 파일명만 봐서는 CV 파이프라인의 일부인지 모른다
* **테스트 경계가 없음** — 모델 교체 시 영향 범위를 추정하기 어렵다

해결은 단순한 물리적 격리였다.

```id="cv-tree"
cv/
├── __init__.py
├── camera_source.py   ← OpenCV/Picamera 추상화
├── face_db.py         ← 얼굴 임베딩 DB
├── vision.py          ← 얼굴 인식 파이프라인
└── models/            ← 온디바이스 모델 가중치
```

이 패키지는 `devices/camera.py` 가 소비하는 형태이다.
즉, CV 레이어는 장치 레이어보다 아래에 있다.

---

### 3.3 `hal/` — Hardware Abstraction Layer

이 레이어는 이번 단계에서 가장 나중에 도입되었으나,
**Pi 이식성** 관점에서 가장 중요한 결정이었다.

```id="hal-tree"
hal/
├── __init__.py
├── display.py       ← 화면 회전 (xrandr / wlr-randr / noop)
├── metrics.py       ← CPU/MEM/DISK/NET + Pi 온도 센서
└── ble_scanner.py   ← BLE 스캔 다이얼로그 (bleak + asyncio)
```

가장 상징적인 변화는 **`run.py` 가 54줄 짧아진 것**이다.
`psutil` 직접 호출, `xrandr` 서브프로세스, BLE 스캔 로직이
전부 HAL 로 옮겨졌다.

```id="run-diff"
- run.py: CPU 온도 센서 키를 직접 탐색
+ run.py: hal.metrics.collect() 한 줄
```

이 분리의 의미는 다음과 같다:

* 개발 PC(x86) → Pi 5 이식 시 **`hal/` 만 열면 된다**
* 테스트 시 HAL 을 mock 하면 UI 로직을 독립 검증할 수 있다
* `hal/` 내부는 OS 를 감지해서 적절한 백엔드를 고른다 (X11 / Wayland / headless)

## 4. The Plugin Workflow

이번 설계에서 가장 공들인 부분은
**"사용자가 새 장치를 추가하는 경험"** 이다.

### 4.1 기존 워크플로 (Diary #1 시점)

```id="old-flow"
1. run.py 열기
2. 새 패널 클래스 작성
3. DeviceStatusWidget 의 분기 수정
4. 재시작
```

→ UI 파일을 건드려야 장치가 늘어난다는 문제.

---

### 4.2 새로운 워크플로

```id="new-flow"
1. 앱의 [⚙ CONFIG] 클릭
2. 장치 타입 입력 (예: "led_panel")
3. 저장 → devices/led_panel.py 스텁 자동 생성
4. 에디터에서 구현만 채워넣기
5. 재시작
```

스텁 생성은 레지스트리의 `generate_device_stub()` 가 담당한다.

생성되는 파일은 다음을 포함한다:

* `@register("led_panel")` 데코레이터가 붙은 `LedPanelDevice` 클래스
* `Device` ABC 의 필수 메서드 스켈레톤 (`from_config`, `make_panel`)
* 동일 파일 안의 `LedPanelPanel` 위젯 (중앙 패널용)
* 테마 토큰 (`ACCENT_TEAL`, `TEXT_BODY` 등) 가 이미 import 되어 있음

또한 `devices/__init__.py` 에
`from . import led_panel` 한 줄이 **자동 append** 된다.

즉, 사용자가 수정할 파일은 **스텁 한 개뿐**이다.

---

### 4.3 Config 스키마 검증

`run.py::_validate_device_cfg()` 는 `config.json` 의 각 엔트리를 검증한다.

```id="config-schema"
{
  "id":    str (non-empty),
  "type":  str (non-empty),
  "label": str (non-empty),
  "kind":  "bluetooth" | "serial" | "camera" | "other"
}
```

잘못된 엔트리는 **스킵하고 경고 로그**를 남긴다.
UI 단계에서 전체가 죽지 않도록 하는 것이 목적이다.

`status` 필드는 더 이상 config 에 저장되지 않는다.
**런타임 상태는 런타임이 소유한다** — 저장 시 제거된다.

## 5. Design Principles Refined

Diary #1 에서 정리했던 원칙들은 유지된다.
이번 단계에서는 여기에 세 가지를 추가로 명문화했다.

### 5.1 "UI Contract is Frozen"

* 상태 상수 5개 (`NOT_IMPLEMENTED` / `DISCONNECTED` / `INITIALIZING` / `OK` / `ERROR`)
* 3-패널 레이아웃 + 하단 감정 레이어
* Miku 카오모지 상태 테이블

이 계약은 **런타임이 어떻게 바뀌든** 깨지지 않는다.
즉, 장치 레이어가 바뀌어도 UI는 동일하게 렌더링된다.

---

### 5.2 "Configuration Over Code"

장치 목록은 `config.json` 에서 선언적으로 관리된다.

```id="config-example"
{
  "devices": [
    { "id": "desk_fan",  "type": "smart_fan", "label": "Smart Fan", "kind": "serial" },
    { "id": "camera_1",  "type": "camera",    "label": "Camera",    "kind": "camera", "backend": "auto", "index": 0 }
  ]
}
```

사용자는 **코드를 수정하지 않고** 장치를 추가/제거할 수 있다.
코드는 오직 **새로운 장치 *타입*** 이 필요할 때만 건드린다.

---

### 5.3 "Isolate What Changes for Different Reasons"

* UI 컴포넌트 (`run.py` 의 `GlassPanel`, `PulseDot`, `CircleGauge`) — 디자인 변경 시
* 장치 플러그인 (`devices/*.py`) — 신규 IoT 기기 추가 시
* CV 파이프라인 (`cv/*.py`) — 모델 교체/튜닝 시
* HAL (`hal/*.py`) — OS / 하드웨어 이식 시

각 레이어는 서로 다른 이유로 변경된다.
따라서 서로 다른 파일/패키지로 분리되어야 한다.

---

## 6. Key Challenges Encountered

### 6.1 "Where does `face_db` belong?"

얼굴 DB 는 `cv/` 에 속하지만,
`devices/camera.py` 와 `config_dialog.py` (얼굴 등록 UI) 가 **공유**한다.

처음에는 각자 인스턴스화하게 두었으나,
얼굴 DB 가 **두 벌** 로드되면서 동기화가 깨졌다.

현재 해결책:

* `JarvisWindow` 가 **단일 FaceDB 인스턴스를 소유** 한다
* 장치 config 에 `_face_db` 키로 **컨텍스트 주입** 한다
* Config 다이얼로그에도 동일 인스턴스를 넘긴다

이는 "소유권은 최상위가 가지되, 의존성은 아래로 주입한다" 는
클래식한 IoC 패턴의 재발견이었다.

---

### 6.2 "What happens when a device type is unknown?"

사용자가 `config.json` 을 직접 편집해서
레지스트리에 없는 `type` 을 적었을 때의 동작을 정의해야 했다.

선택지:

1. **앱 크래시** — 명확하지만 가혹하다
2. **엔트리 무시** — 조용하지만 디버깅이 어렵다
3. **스킵 + 경고 로그 + CenterPanel 기본 페이지 노출** — 채택

레지스트리의 `build()` 는
미등록 타입에 대해 `None` 을 반환하고 로그만 남긴다.
UI 는 계속 동작하므로 사용자는 *무엇이 잘못되었는지* 눈으로 볼 수 있다.

---

### 6.3 "How much boilerplate should stubs carry?"

스텁이 너무 비면 사용자가 처음부터 쓰는 것과 다를 바 없다.
반대로 너무 차 있으면 사용자는 *지울 코드* 부터 읽어야 한다.

현재 스텁이 포함하는 것:

* 등록 데코레이터 + ABC 메서드 3개 (필수)
* `QVBoxLayout` + `QLabel("{label}")` 뼈대 (즉시 실행 가능)
* `statusChanged = pyqtSignal(str, str)` (UI 계약 준수용)
* 테마 토큰 import (`ACCENT_TEAL`, `TEXT_BODY`, `S`)

포함하지 **않는** 것:

* 통신 로직 — 장치마다 다르므로 주석 `# TODO` 로만 남긴다
* 상태 전이 다이어그램 — 구현자가 장치 특성에 맞춰 작성한다

결과적으로 **"앱이 죽지 않고 뜨는 최소 구현"** 을 제공한다.

## 8. Conclusion

Diary #1 은 UI의 **겉** 을 그렸다.
Diary #2 는 UI의 **속** 을 뜯어고쳤다.

이번 단계에서 도출된 구조적 결론은 다음과 같다.

* **UI 계약은 동결하고, 런타임만 교체한다** — 사용자 경험의 안정성
* **플러그인은 파일 하나로 완결되어야 한다** — 기여 장벽의 최소화
* **OS 의존성은 하나의 레이어에 모은다** — 이식 비용의 사전 격리

결과적으로 프로젝트는 이제
"닫힌 대시보드" 가 아니라 **"확장 가능한 런타임"** 이 되었다.
