# Dev Diary #1 — Character-Augmented Debug Dashboard Design

## 1. Overview

본 프로젝트는 라즈베리파이 기반 환경에서 동작하는
**인터랙티브 AI 인터페이스**를 구축하는 것을 목표로 한다.

최종적으로는 단순한 시스템 모니터링을 넘어,
사용자와 상호작용하는 **캐릭터 기반 인터페이스**를 구현하는 것이 목적이다.

현재 단계에서는 전체 시스템의 기반이 되는
**디버그 대시보드 UI**를 우선적으로 설계 및 구현하고 있다.

---

## 2. Design Objectives

이번 단계의 주요 목표는 다음과 같다:

* IoT 디바이스 상태를 시각적으로 표현할 수 있는 구조 설계
* 시스템 리소스 모니터링 UI 구현
* 향후 AI 및 인터랙션 레이어를 결합하기 위한 UI 기반 마련

특히, 단순한 정보 출력이 아니라
**확장 가능한 인터페이스 구조 설계**에 초점을 맞추었다.

---

## 3. System Layout

현재 UI는 크게 세 개의 영역으로 구성되어 있다.

### 3.1 IoT Device Panel

좌측 패널에는 관리 대상 IoT 디바이스 목록을 배치하였다.

* Camera
* Speaker
* Sensor Hub
* Smart Light
* Door Lock

현재는 실제 디바이스 연결 이전 단계이므로,
모든 상태는 다음과 같이 표현된다:

```id="zqg0ey"
NOT_IMPLEMENTED
```

이 값은 단순한 placeholder가 아니라,
향후 상태 전이(state transition)를 고려한 **명시적 상태 정의**이다.

---

### 3.2 System Metrics Panel

우측 패널에는 시스템 상태를 나타내는 주요 지표를 배치하였다.

* CPU Usage / Temperature
* Memory Usage
* Disk Usage
* Network Throughput

해당 영역은 실제 시스템 데이터를 기반으로 동작하며,
디버깅 시 즉각적인 상태 파악이 가능하도록 구성하였다.

---

### 3.3 Character Status Layer

하단에는 캐릭터 기반 상태 표현 레이어를 추가하였다.

```id="tjy9vd"
MIKU: (｡•‿•｡)
상태: 대기중이에요!
```

이 레이어는 단순한 시각적 요소가 아니라,
향후 AI 모듈과 연결되어 상태 및 이벤트를
**자연어 형태로 표현하는 인터페이스 역할**을 수행하게 된다.

---

## 4. Design Strategy

본 UI의 핵심 설계 방향은 다음과 같다:

> **"High-density debug interface augmented with a minimal character layer"**

---

### 4.1 Separation of Concerns

* 디버그 정보 (정확성, 가독성)
* 캐릭터 표현 (감성, 사용자 경험)

두 요소를 명확히 분리하여,
각각의 목적이 서로를 침해하지 않도록 설계하였다.

---

### 4.2 Minimal Character Injection

캐릭터 요소는 다음과 같은 제약을 가진다:

* UI의 정보 전달을 방해하지 않을 것
* 시각적 노이즈를 최소화할 것
* 텍스트 기반 표현 중심으로 제한할 것

이를 통해 캐릭터는 “주요 콘텐츠”가 아니라,
**보조 인터페이스 레이어**로 기능하도록 한다.

---

### 4.3 Visual Language

* Dark theme 기반
* Teal 계열 색상을 중심으로 한 제한된 컬러 팔레트
* Soft glow (diffused lighting) 적용
* Rounded geometry (8–16px radius)

이를 통해 JARVIS 스타일의 구조를 유지하면서도,
과도한 사이버펑크 톤을 배제하였다.

---

## 5. Key Challenges

### 5.1 Readability vs Expressiveness

캐릭터 요소를 추가할 경우,
다음과 같은 트레이드오프가 발생한다:

* 표현력 증가 ↔ 정보 전달력 감소

현재는 다음과 같은 방식으로 해결하였다:

* 핵심 상태 정보는 유지
* 표현 레이어만 캐릭터화

---

### 5.2 State Representation

초기에는 모든 상태를 `UNKNOWN`으로 표현했으나,
이는 디버깅 관점에서 의미가 모호하다는 문제가 있었다.

따라서 다음과 같은 상태 체계를 도입하였다:

```id="2o2o3p"
NOT_IMPLEMENTED
DISCONNECTED
INITIALIZING
OK
ERROR
```

이를 통해 UI 단계에서도 명확한 상태 의미를 유지할 수 있다.

---

### 5.3 Character Integration Scope

캐릭터를 UI에 어느 수준까지 포함할 것인지에 대한 고민이 있었다.

현재 기준은 다음과 같다:

* 대형 캐릭터 렌더링은 배제
* 소형 애니메이션 또는 텍스트 기반 표현 중심
* UI 구조를 침범하지 않는 범위 내에서 제한적 사용

---

## 6. Next Steps

다음 단계에서는 다음 작업을 진행할 예정이다:

* IoT 디바이스 실제 연결
* 상태 관리 로직 (state machine) 구현
* 중앙 영역에 로그 스트림 추가
* 캐릭터 레이어를 이벤트 기반 반응 구조로 확장

---

## 7. Conclusion

본 단계에서는 단순한 UI 구현을 넘어,
**확장 가능한 인터페이스 구조 설계**에 집중하였다.

특히, 디버그 UI와 캐릭터 인터페이스를 결합하는 과정에서
다음과 같은 설계 원칙을 도출할 수 있었다:

* 기능성과 감성은 분리하되, 충돌하지 않게 통합할 것
* 캐릭터는 콘텐츠가 아니라 인터페이스로 다룰 것
* 상태 표현은 초기 단계부터 명확하게 정의할 것

이 구조를 기반으로, 향후
**상호작용 가능한 AI 시스템**으로 확장할 계획이다.
